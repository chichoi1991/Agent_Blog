## Copilot Studio에 적합한 요구사항

아래 조건에 해당하는 프로젝트는 Copilot Studio만으로 충분히 구현 가능합니다.

### 대화형 안내 및 FAQ Agent

- 사내 정책, 제품 정보, 자주 묻는 질문에 대한 **자동 응답**
- SharePoint, 웹사이트 등에 이미 문서화된 정보를 기반으로 한 **검색·요약**
- **일회성 정보 조회** 위주의 시나리오 (예: "연차 잔여일 확인", "출장비 신청 양식 안내")

### 간단한 업무 자동화

- 이메일 발송, Teams 메시지 전송, SharePoint 목록 등록 등 **표준 커넥터**로 처리 가능한 작업
- Power Automate 플로우와 연계한 **승인 요청, 알림 전달**
- **3단계 이하의 분기 처리**로 충분한 업무 흐름

### 빠른 PoC / MVP 구축

- 아이디어를 빠르게 검증해야 하는 **프로토타이핑** 단계
- 개발자 투입 없이 **현업 담당자가 직접** 구축·수정할 수 있는 범위
- **수일~수주** 단위로 결과물을 확인해야 하는 프로젝트

### 적합성 체크리스트

| 항목 | 기준 |
|---|---|
| 대화 복잡도 | 단순 FAQ, 선형 흐름, 3단계 이하 분기 |
| 데이터 소스 | SharePoint, 웹사이트, Dataverse, 표준 커넥터 |
| 사용자 수 | 수백~수천 명 (조직 내부) |
| UI 요구사항 | 기본 채팅 UI, Adaptive Card |
| 유지보수 주체 | 현업/시민개발자 |
| 인증 방식 | Entra ID 기본 통합 |

## 직접 개발(SI)이 필요한 요구사항

아래 조건이 하나 이상 해당되면, Copilot Studio 단독으로는 한계가 있으며 직접 개발을 고려해야 합니다.

### 복잡한 비즈니스 로직

- **다단계 트랜잭션**: 복수 시스템 간 데이터 정합성이 보장되어야 하는 처리
- **동적 워크플로우**: 사용자 역할, 조직 구조, 시간 조건 등에 따라 흐름이 크게 달라지는 경우
- **대규모 데이터 처리**: 수만 건 이상의 데이터를 실시간으로 집계·분석해야 하는 경우

### 고도화된 UI/UX 요구

- 채팅 UI 외에 **대시보드, 차트, 커스텀 폼** 등이 필요한 경우
- 사용자 경험을 세밀하게 제어해야 하는 **B2C 서비스**
- **키오스크, 모바일 네이티브 앱** 등 특수 디바이스 대응

### 레거시 시스템 직접 연동

- **사내 DB(Oracle, SQL Server 등)에 직접 쿼리**해야 하는 경우
- 표준 커넥터가 없는 **사내 전용 시스템**(ERP, PLM 등)과의 연동
- **VPN/방화벽 내부** 시스템에 접근해야 하는 경우 (On-Premises Data Gateway 활용 가능하나 제약 있음)

### 엄격한 보안/규정 준수

- **커스텀 인증 흐름** (mTLS, SAML, 커스텀 토큰 발급 등)
- **데이터 주권** 요구로 클라우드 SaaS 사용이 불가한 경우
- 감사(Audit) 로그, 접근 이력 추적 등에 대한 **세밀한 제어**가 필요한 경우

## 혼합 구조 (Studio + 개발) 패턴

실무에서는 Copilot Studio와 직접 개발을 **함께 사용하는 혼합 구조**가 가장 현실적인 선택인 경우가 많습니다.

### 패턴 1: Studio 프론트 + Azure 백엔드

```
[사용자] → [Copilot Studio Agent]
                 ↓
         [Power Automate 플로우]
                 ↓
         [Azure Functions / Logic Apps]
                 ↓
         [사내 DB / 외부 API]
```

- **대화 인터페이스**는 Copilot Studio가 담당
- **복잡한 비즈니스 로직**은 Azure Functions으로 분리
- Power Automate가 **중간 연결 계층** 역할

**적합 시나리오**: 대화형 인터페이스가 필요하지만, 내부적으로 복잡한 데이터 처리가 있는 경우

### 패턴 2: Studio + MCP 서버

```
[사용자] → [Copilot Studio Agent]
                 ↓ (MCP Protocol)
         [자체 MCP 서버]
                 ↓
         [IoT 디바이스 / 사내 시스템 / 외부 API]
```

- MCP 서버를 직접 개발하여 **Agent의 도구(Tool)로 제공**
- Agent는 지침에 따라 MCP 도구를 **자율적으로 호출**
- 새로운 도구 추가 시 MCP 서버만 업데이트하면 됨

**적합 시나리오**: 다양한 외부 시스템을 유연하게 연동해야 하는 경우

### 패턴 3: Studio + Custom Engine Agent

```
[사용자] → [Copilot Studio 오케스트레이션]
                 ↓
         [Custom Engine Agent (자체 LLM 엔진)]
                 ↓
         [특화 모델 / RAG 파이프라인 / 벡터 DB]
```

- 대화 채널 관리와 배포는 Copilot Studio가 담당
- **AI 엔진(LLM, RAG)** 은 자체 개발한 Custom Engine Agent가 처리
- Teams, 웹 등 **멀티 채널 배포**의 이점을 유지하면서 AI 성능을 극대화

**적합 시나리오**: 특화된 도메인 지식이나 자체 학습 데이터가 필요한 경우

### 의사결정 플로우차트

```
요구사항 분석
  ├─ 대화 복잡도 낮음 + 표준 커넥터 충분 → ✅ Copilot Studio 단독
  ├─ 복잡한 로직 필요 + 대화 UI 필요 → 🔀 패턴 1 (Studio + Azure)
  ├─ 다양한 외부 도구 연동 필요 → 🔀 패턴 2 (Studio + MCP)
  ├─ 자체 AI 엔진 필요 → 🔀 패턴 3 (Studio + Custom Engine)
  └─ 커스텀 UI + 온프레미스 전용 → ❌ 직접 개발(SI)
```

<div class="info-box tip">
<strong>💡 실무 팁</strong>
혼합 구조를 선택할 때는 <strong>"유지보수 주체"</strong>를 먼저 결정하세요. 현업이 직접 관리할 부분은 Studio로, 전문 개발팀이 관리할 부분은 코드로 분리하면, 장기적으로 운영 효율이 높아집니다.
