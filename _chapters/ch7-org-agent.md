## 개인 Agent와 조직 Agent 차이

Copilot Studio에서 만드는 Agent는 사용 범위에 따라 크게 **개인 Agent**와 **조직 Agent**로 구분됩니다.

### 비교표

| 구분 | 개인 Agent | 조직 Agent |
|---|---|---|
| **목적** | 개인 업무 생산성 향상 | 팀/부서/전사 공통 업무 지원 |
| **작성자** | 개인 (시민개발자) | 전담 팀 또는 지정 담당자 |
| **관리 환경** | 개인 환경 또는 기본 환경 | 전용 운영 환경(Production) |
| **Knowledge 소스** | 개인 문서, 소규모 데이터 | 공식 사내 문서, 정책, 데이터 |
| **배포 범위** | 본인 또는 소수 공유 | 특정 부서 또는 조직 전체 |
| **거버넌스** | 느슨함 | 엄격한 승인/검토 프로세스 |
| **유지보수** | 작성자 본인 | 전담 운영팀 (IT/CoE) |
| **SLA** | 없음 | 운영 수준 보장 필요 |

<div class="info-box note">
<strong>📌 참고</strong>
개인 Agent는 개인의 업무 생산성을 위한 도구이고, 조직 Agent는 <strong>공식적인 업무 채널</strong>로 기능합니다. 두 유형은 목적과 관리 체계가 다르므로, 처음부터 용도를 명확히 구분하여 설계하는 것이 중요합니다.
</div>

## 조직 Agent 등록 프로세스

조직 공용 Agent를 공식적으로 운영하려면, 아래와 같은 등록·승인 프로세스를 거치는 것이 권장됩니다.

### 등록 절차 (권장)

```
1. 요구사항 정의 및 기획
   ↓
2. PoC / 프로토타입 제작
   ↓
3. 내부 검토 및 승인
   ↓
4. 운영 환경 배포
   ↓
5. 조직 내 공지 및 교육
   ↓
6. 운영 모니터링 및 개선
```

### Step 1: 요구사항 정의 및 기획

| 항목 | 내용 |
|---|---|
| **대상 업무** | Agent가 처리할 업무 범위 명확화 |
| **대상 사용자** | 사용 대상 그룹 (전사 / 특정 부서) |
| **Knowledge 소스** | 참조할 공식 문서·데이터 목록 |
| **외부 연동** | 필요한 커넥터/API 목록 |
| **관리 담당자** | Agent 운영·유지보수 담당 지정 |
| **성공 기준(KPI)** | 도입 효과 측정 지표 (응답률, 만족도 등) |

### Step 2: PoC / 프로토타입

- 개발 환경에서 Agent를 생성하고 핵심 시나리오를 구현합니다.
- 소규모 사용자 그룹과 함께 **파일럿 테스트**를 진행합니다.
- 피드백을 반영하여 지침, Knowledge, Action을 수정합니다.

### Step 3: 내부 검토 및 승인

- **IT/보안팀 검토**: Knowledge 소스의 보안 적합성, 인증 방식, 데이터 유출 위험
- **법무/컴플라이언스**: 개인정보 처리, 규정 준수 여부
- **관리자 승인**: Power Platform 환경 관리자의 배포 승인

### Step 4: 운영 환경 배포

- 운영(Production) 환경에 Agent를 이전(또는 재구축)합니다.
- Teams Admin Center에서 조직 앱으로 등록합니다.
- 필요 시 **자동 설치 정책**을 통해 대상 사용자에게 자동 배포합니다.

### Step 5: 공지 및 교육

- 조직 내 공지 채널(메일, Teams, 인트라넷)을 통해 Agent 출시를 안내합니다.
- 사용 가이드 및 FAQ를 배포합니다.

## 전사 공용 Agent 운영 시 유의사항

### 거버넌스 (Governance)

| 영역 | 권장 사항 |
|---|---|
| **소유권** | Agent별 소유 팀/담당자를 명확히 지정 |
| **변경 관리** | 지침·Knowledge 변경 시 검토→승인→게시 프로세스 적용 |
| **접근 제어** | 편집 권한(Maker)은 최소 인원에게만 부여 |
| **환경 분리** | 개발 / 테스트 / 운영 환경을 분리하여 관리 |

### 모니터링 및 분석

Copilot Studio의 **Analytics 탭**에서 다음 지표를 정기적으로 모니터링합니다.

| 지표 | 설명 |
|---|---|
| **총 세션 수** | 일정 기간 내 Agent와의 대화 횟수 |
| **해결률(Resolution Rate)** | 사용자 질문을 성공적으로 해결한 비율 |
| **에스컬레이션 비율** | 상담원 전환이 발생한 비율 |
| **CSAT (만족도)** | 사용자 만족도 평가 점수 |
| **미해결 질문** | 답변하지 못한 질문 목록 → Knowledge/Topic 보강 소스 |

### 버전 관리

- Agent의 주요 변경 사항은 **변경 이력**으로 관리합니다.
- 문제 발생 시 이전 버전으로 롤백할 수 있도록, 주기적으로 **Solution 내보내기**를 수행합니다.
- Power Platform의 **Solution** 기능을 활용하면, Agent를 패키지로 관리하고 환경 간 이동할 수 있습니다.

### 콘텐츠 최신성 유지

- Knowledge 소스(SharePoint 문서 등)가 최신 상태인지 **주기적으로 검증**합니다.
- 정책 변경, 조직 개편 등이 있을 때 지침과 Knowledge를 **즉시 업데이트**합니다.
- 정기 리뷰 일정을 설정하여 Agent의 응답 품질을 **지속적으로 개선**합니다.

## 조직 Agent 활용 예시

### (예) HR 가이드 Agent

| 항목 | 설정 |
|---|---|
| **역할** | 인사 정책 안내 전문 Agent |
| **Knowledge** | SharePoint의 인사 정책 문서, 복리후생 안내, 취업규칙 |
| **주요 Topic** | 연차 신청 방법, 출장비 정산, 복리후생 문의 |
| **Actions** | 잔여 연차 조회 (HR 시스템 API), 경조사 신청 (Power Automate) |
| **배포 채널** | Teams (전사 배포) |
| **관리 담당** | 인사팀 + IT CoE 공동 운영 |

**사용자 시나리오:**

```
사용자: "올해 연차 몇 일 남았어?"
Agent: HR 시스템에서 조회한 결과, 홍길동 님의 잔여 연차는 8일입니다.
       [연차 신청하기] [연차 정책 보기]
```

### (예) IT 헬프데스크 Agent

| 항목 | 설정 |
|---|---|
| **역할** | IT 장애·요청 접수 및 자가 해결 안내 Agent |
| **Knowledge** | SharePoint의 IT FAQ, 매뉴얼, 설치 가이드 |
| **주요 Topic** | 비밀번호 초기화, VPN 연결, 프린터 설정 |
| **Actions** | IT 티켓 생성 (ServiceNow API), 비밀번호 리셋 요청 (Power Automate) |
| **배포 채널** | Teams (전사 배포) + 사내 포털 (웹 임베드) |
| **관리 담당** | IT 인프라팀 |

**사용자 시나리오:**

```
사용자: "비밀번호를 잊어버렸어요"
Agent: 비밀번호 초기화를 도와드리겠습니다.
       먼저 사번을 입력해주세요.
       
사용자: "20230001"
Agent: 사번 20230001로 비밀번호 초기화 요청을 접수했습니다.
       등록된 이메일로 안내 메일이 발송됩니다.
       [IT 헬프데스크 문의하기]
```

<div class="info-box tip">
<strong>💡 실무 팁</strong>
조직 공용 Agent를 처음 도입할 때는 <strong>하나의 업무 영역</strong>에 집중하여 시작하세요. "모든 것을 다 해결하는 Agent"보다, "특정 업무를 확실하게 처리하는 Agent"가 사용자 신뢰를 빠르게 확보합니다. 성공 사례를 만든 후 다른 업무 영역으로 확장하는 것이 효과적입니다.
</div>
